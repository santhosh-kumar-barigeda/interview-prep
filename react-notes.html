Chapter 1

Introduction to React

What is React?

React is a JavaScript library for building user interfaces. It was created by Facebook (Meta) and is component-based — you build small, reusable pieces of UI that compose together into complex applications.

  • Declarative — describe what the UI should look like for a given state, not how to update it. React figures out the DOM changes.
  • Component-based — UI is split into self-contained components, each managing its own logic and appearance.
  • Unidirectional data flow — data flows down from parent to child via props. Events flow up via callbacks.
  • Virtual DOM — React keeps a lightweight copy of the DOM in memory. On state changes it diffs the virtual DOM and applies only the minimal real DOM updates.
React is a library, not a framework. It only handles the view layer. Routing, data fetching, and state management are handled by additional libraries (React Router, TanStack Query, Zustand, etc.) or a meta-framework like Next.js.

How React Works — The Layers

When you write React, your code passes through several layers before anything appears on screen:

LayerWhat it does
Your componentsPlain JS functions returning JSX — describe the UI for a given state.
React (reconciler)Calls your components, builds a virtual tree, diffs it against the previous tree, decides what changed.
react-dom (renderer)Takes the reconciler's diff output and applies the minimal changes to the real browser DOM.
Browser DOMThe actual HTML elements the user sees. React manages this for you.
react and react-dom are separate packages on purpose. react is the platform-agnostic reconciler. react-dom is just one renderer — the same reconciler also powers react-native (iOS/Android), react-three-fiber (3D/WebGL), and more.

Virtual DOM

The Virtual DOM is a lightweight in-memory JavaScript object tree that mirrors the real DOM. When your component returns JSX, React creates plain JS objects (Virtual DOM nodes). On every state change React builds a new virtual tree, diffs it against the previous one, and applies only the necessary real DOM operations — avoiding redundant browser reflows.

// JSX you write <div className="card"><p>Hello</p></div> // Babel compiles it to React.createElement calls React.createElement("div", { className: "card" }, React.createElement("p", null, "Hello") ) // Which produces a plain JS object — the Virtual DOM node { type: "div", props: { className: "card" }, children: [{ type: "p", ... }] }

Fiber & the Render / Commit Phases

Fiber is React's internal reconciler engine (since React 16). It represents every component as a Fiber node — a linked-list unit of work — so React can process the tree incrementally and pause mid-way to handle higher-priority updates (e.g. a keystroke interrupting a slow list render).

Every update goes through two phases:

PhaseWhat happensKey rule
Render phaseReact calls your components, builds the new Fiber tree, diffs it, produces an effects list. No DOM is touched yet.Must be pure — can be paused or restarted in Concurrent Mode.
Commit phaseReact applies all DOM mutations in one synchronous pass, attaches refs, then fires useLayoutEffect and schedules useEffect.
// Full timeline for one state update: setCount(c => c + 1); // 1. you call setState // 2. React schedules work (batches with other updates) // 3. Render phase — component re-runs (pure, no DOM changes yet) // 4. Reconciliation — diff: "text in <p> changed from 0 → 1" // 5. Commit phase — p.textContent = "1" (real DOM updated) // 6. Browser paints // 7. useLayoutEffect fires (sync, before paint is visible) // 8. useEffect fires (async, after paint)

State Batching & Concurrent Mode

Batching — React groups multiple state updates that happen together into a single re-render. In React 18 this is automatic everywhere (including inside setTimeout and fetch callbacks).

function handleClick() { setCount(c => c + 1); // ─┐ setName("Alice"); // ├── one re-render, not three setLoading(false); // ─┘ }

Concurrent Mode (React 18+) lets React prepare multiple versions of the UI at once and interrupt low-priority work for urgent updates — keeping the UI responsive without you changing your component code.

APIWhat it does
useTransitionMark a state update as non-urgent. React keeps the current UI while building the next one in the background.
useDeferredValueDefer re-rendering a slow part of the UI while fast parts update immediately.
SuspenseShow a fallback while async content (data, lazy components) is loading, then swap it in when ready.

Reconciliation & Keys

React's reconciliation algorithm compares the old and new virtual DOM trees. It uses element type and position to decide whether to reuse a DOM node or recreate it.

// Type change — React destroys and recreates (loses state) {isEditing ? <input /> : <p>Text</p>} // Same type, same position — React REUSES the DOM node and updates props {isActive ? <Button color="green" /> : <Button color="red" />} // Force a fresh mount by changing the key <EmailInput key={userId} /> {/* new key = new instance = reset state */}
Triggers for re-render: (1) setState / dispatch called, (2) a context value the component consumes changes, (3) the parent re-renders and passes new props. React re-renders the component and all its descendants by default — use React.memo to opt out.

Strict Mode

<React.StrictMode> wraps your app (or part of it) to enable additional development-only checks. It intentionally double-invokes render functions and effects to help detect side effects in the render phase.

import { StrictMode } from "react"; createRoot(document.getElementById("root")).render( <StrictMode> <App /> </StrictMode> );
In development, Strict Mode mounts every component twice to expose effects that aren't properly cleaned up. If you see effects running twice, make sure your useEffect returns a cleanup function. This double-invoke does not happen in production.

Core Concepts

JSX

JSX (JavaScript XML) is a syntax extension that lets you write HTML-like markup inside JavaScript. Babel/bundlers transform JSX into React.createElement() calls at build time — it is not a separate template language, just syntactic sugar over plain JS function calls.

// JSX — what you write const el = <h1 className="title">Hello, {name}</h1>; // What Babel compiles it to const el = React.createElement("h1", { className: "title" }, `Hello, ${name}`);

No statements in { }Can't use if, for — use ternary or && instead
camelCase event handlersonClick, onChange, onSubmit

function Card({ title, count, isNew }) { return ( <> {/* Fragment — no extra DOM node */} <h2 className="card-title">{title}</h2> {isNew && <span className="badge">New</span>} {/* conditional render */} <p>Count: {count * 2}</p> {/* expression */} <input type="text" /> {/* self-closing */} </> ); }

Function Components

A component is a JavaScript function that accepts props as its argument and returns JSX. Component names must start with a capital letter — React uses this to distinguish components from native HTML elements.

// Simplest component function Hello() { return <h1>Hello, world</h1>; } // Component with props function Button({ label, onClick, disabled = false }) { return ( <button onClick={onClick} disabled={disabled}> {label} </button> ); } // Usage — JSX looks like HTML attributes <Button label="Save" onClick={handleSave} /> <Button label="Delete" disabled /> {/* boolean prop — just the name = true */}

State

State is data owned by the component that can change over time. When state changes, React re-renders the component and its children. Unlike props (which come from outside and are read-only), state is internal and mutable via its setter function.

PropsState
Owned byParentThe component itself
Mutable?No — read-onlyYes — via setter
Triggers re-render?Yes, when parent re-rendersYes, when setter is called

Props

Props (properties) are the inputs to a component — data passed from parent to child. Props are read-only; a component must never modify its own props.

// Parent passes data down function UserCard({ name, avatar, role = "member" }) { return ( <div className="user-card"> <img src={avatar} alt={name} /> <h3>{name}</h3> <p>{role}</p> </div> ); } // Pass anything as a prop — strings, numbers, objects, functions, JSX <UserCard name="Alice" avatar="/img/alice.jpg" role="admin" /> // Spread an object as props const user = { name: "Bob", avatar: "/bob.jpg" }; <UserCard {...user} />

children prop

Whatever JSX is placed between a component's opening and closing tags is passed as the special children prop. This lets you build wrapper/layout components.

function Card({ children, className = "" }) { return ( <div className={`card ${className}`}> {children} </div> ); } // Usage — anything between the tags becomes children <Card className="featured"> <h2>Title</h2> <p>Some content here.</p> </Card>
Chapter 3

Hooks

Rules of Hooks

  • Only call hooks at the top level — never inside loops, conditions, or nested functions. This ensures React calls hooks in the same order on every render.
  • Only call hooks from React functions — function components or custom hooks. Not from regular JS functions, class components, or event handlers.
// ❌ Conditional hook — breaks ordering if (condition) { const [x, setX] = useState(0); // never do this } // ✅ Hook always called — condition inside const [x, setX] = useState(0); if (condition) { setX(1); }

useState

useState returns a pair: the current state value and a setter function. The setter triggers a re-render with the new value.

import { useState } from "react"; function Counter() { const [count, setCount] = useState(0); // 0 is the initial value return ( <div> <p>Count: {count}</p> <button onClick={() => setCount(count + 1)}>+</button> <button onClick={() => setCount(count - 1)}>-</button> <button onClick={() => setCount(0)}>Reset</button> </div> ); }

Functional Updater Form

When the new state depends on the previous state, use the functional updater: pass a function to the setter instead of a value. This guarantees you get the latest state — important when updates are batched or happen in async code.

// ❌ May use stale state if called multiple times quickly setCount(count + 1); setCount(count + 1); // both see the same `count` — only increments by 1 // ✅ Functional updater — always receives latest state setCount(prev => prev + 1); setCount(prev => prev + 1); // correctly increments by 2

State with Objects & Arrays

State must be treated as immutable. Never mutate state directly — always create a new object or array and pass that to the setter. React uses shallow comparison to detect changes.

const [user, setUser] = useState({ name: "Alice", age: 25 }); // ❌ Direct mutation — React won't detect the change user.age = 26; setUser(user); // same reference — no re-render // ✅ Spread to create new object setUser({ ...user, age: 26 }); // Arrays — use non-mutating methods const [items, setItems] = useState(["a", "b"]); setItems([...items, "c"]); // add setItems(items.filter(i => i !== "a")); // remove setItems(items.map(i => i === "b" ? "B" : i)); // update

useEffect

useEffect lets you run side effects after a component renders. Side effects are anything that reaches outside React's rendering — fetching data, setting up subscriptions, timers, manually updating the DOM, etc.

import { useEffect } from "react"; // Runs after every render useEffect(() => { document.title = `Count: ${count}`; }); // Runs once on mount (empty dependency array) useEffect(() => { fetchUser(); }, []); // Runs when `userId` changes (and on mount) useEffect(() => { fetchUser(userId); }, [userId]); // With cleanup — returned function runs before next effect & on unmount useEffect(() => { const id = setInterval(() => tick(), 1000); return () => clearInterval(id); // cleanup }, []);
The dependency array controls when the effect runs:
— No array → runs after every render.
— [] → runs once on mount, cleanup on unmount.
— [a, b] → runs when a or b changes (and on mount).

Always include every reactive value used inside the effect in the dependency array. ESLint's exhaustive-deps rule enforces this.

Fetching Data with useEffect

function UserProfile({ userId }) { const [user, setUser] = useState(null); const [loading, setLoading] = useState(true); const [error, setError] = useState(null); useEffect(() => { let cancelled = false; // prevent stale response on fast prop changes setLoading(true); fetch(`/api/users/${userId}`) .then(r => r.json()) .then(data => { if (!cancelled) setUser(data); }) .catch(err => { if (!cancelled) setError(err); }) .finally(() => { if (!cancelled) setLoading(false); }); return () => { cancelled = true; }; // cleanup on userId change or unmount }, [userId]); if (loading) return <p>Loading…</p>; if (error) return <p>Error: {error.message}</p>; return <h2>{user.name}</h2>; }
In real projects, prefer a data-fetching library (TanStack Query, SWR) over raw useEffect for data fetching. They handle caching, deduplication, loading/error states, and refetching automatically.

useRef

useRef returns a mutable ref object whose .current property persists across renders. It has two main uses:

DOM access

Attach to a JSX element with ref={myRef} to get a direct reference to the DOM node. Available after mount.

Mutable value

Store any value that should persist across renders without triggering a re-render when changed. Unlike state, updating .current is silent.

import { useRef } from "react"; // DOM reference function FocusInput() { const inputRef = useRef(null); return ( <> <input ref={inputRef} /> <button onClick={() => inputRef.current.focus()}>Focus</button> </> ); } // Mutable value — tracking previous state without re-render function Timer() { const intervalRef = useRef(null); const start = () => { intervalRef.current = setInterval(tick, 1000); }; const stop = () => { clearInterval(intervalRef.current); }; }

useMemo

useMemo caches (memoizes) the result of an expensive computation. It only recalculates when one of its dependencies changes. Use it to avoid repeating heavy work on every render.

import { useMemo } from "react"; function ProductList({ products, searchTerm }) { // Only re-filters when products or searchTerm changes const filtered = useMemo( () => products.filter(p => p.name.includes(searchTerm)), [products, searchTerm] ); return filtered.map(p => <ProductCard key={p.id} {...p} />); }
Don't overuse useMemo. Memoization itself has a cost. Only use it when you have a genuinely expensive computation or when referential stability of the value matters (e.g. passing it to a child wrapped in React.memo).

useCallback

useCallback returns a memoized version of a callback function. The function is only recreated when its dependencies change. Without it, a new function reference is created on every render, which can cause children to re-render unnecessarily.

import { useCallback } from "react"; function Parent() { const [count, setCount] = useState(0); // Without useCallback — new function on every render const handleClick = () => setCount(c => c + 1); // With useCallback — same reference across renders (deps don't change) const handleClick = useCallback( () => setCount(c => c + 1), [] // no deps — function never changes ); return <Child onClick={handleClick} />; }

useContext

useContext reads the value from a React context. Context lets you pass data deep into the component tree without passing props manually at every level (prop drilling).

import { createContext, useContext, useState } from "react"; // 1. Create context with a default value const ThemeContext = createContext("light"); // 2. Provide the value at the top of the tree function App() { const [theme, setTheme] = useState("dark"); return ( <ThemeContext.Provider value={theme}> <Page /> </ThemeContext.Provider> ); } // 3. Consume anywhere in the tree — no prop drilling needed function Button() { const theme = useContext(ThemeContext); return <button className={theme}>Click</button>; }
Context re-renders all consumers when the value changes. For frequently updating values (e.g. mouse position), context can be a performance problem. Use it for slowly-changing globals: theme, locale, auth user, feature flags.

useReducer

useReducer is an alternative to useState for complex state that involves multiple sub-values or where the next state depends on the previous in non-trivial ways. It uses a reducer function — a pure function of (state, action) => newState.

import { useReducer } from "react"; function reducer(state, action) { switch (action.type) { case "increment": return { ...state, count: state.count + 1 }; case "decrement": return { ...state, count: state.count - 1 }; case "reset": return { count: 0 }; default: throw new Error(`Unknown action: ${action.type}`); } } function Counter() { const [state, dispatch] = useReducer(reducer, { count: 0 }); return ( <> <p>{state.count}</p> <button onClick={() => dispatch({ type: "increment" })}>+</button> <button onClick={() => dispatch({ type: "reset" })}>Reset</button> </> ); }

useLayoutEffect

useLayoutEffect has the same signature as useEffect but fires synchronously after all DOM mutations and before the browser paints. Use it when you need to read layout (e.g. element dimensions) and synchronously re-render before the user sees anything — avoids the flicker that useEffect would cause.

import { useLayoutEffect, useRef, useState } from "react"; function Tooltip({ children }) { const ref = useRef(null); const [height, setHeight] = useState(0); // Runs after DOM update but BEFORE browser paint — no flicker useLayoutEffect(() => { setHeight(ref.current.getBoundingClientRect().height); }, []); return <div ref={ref}>{children}</div>; }
Prefer useEffect by default. useLayoutEffect blocks the browser from painting — use it only when you need to measure the DOM before the user sees the render.

useId

useId() generates a stable, unique ID that is consistent between server and client renders. Use it for accessibility attributes like htmlFor, aria-labelledby, and aria-describedby — never for list keys.

import { useId } from "react"; function EmailField() { const id = useId(); // e.g. ":r1:" return ( <> <label htmlFor={id}>Email</label> <input id={id} type="email" /> </> ); }

useTransition

useTransition() returns [isPending, startTransition]. Wrap a state update in startTransition to mark it as non-urgent — React will keep the current UI responsive and process the transition in the background. isPending is true while the transition is in progress.

import { useTransition, useState } from "react"; function SearchPage() { const [query, setQuery] = useState(""); const [results, setResults] = useState([]); const [isPending, startTransition] = useTransition(); function handleChange(e) { setQuery(e.target.value); // urgent — updates input immediately startTransition(() => { setResults(search(e.target.value)); // non-urgent — can be deferred }); } return ( <> <input value={query} onChange={handleChange} /> {isPending ? <p>Searching…</p> : <ResultList items={results} />} </> ); }

useDeferredValue

useDeferredValue(value) returns a deferred copy of the value that lags behind the real value during high-priority updates. React updates the deferred value in the background after urgent updates (like typing) are processed. Useful for keeping a fast input while a slow list re-renders.

import { useDeferredValue, useState } from "react"; function FilteredList({ items }) { const [query, setQuery] = useState(""); const deferredQuery = useDeferredValue(query); // lags behind query // Re-renders with deferredQuery — won't block the input const filtered = items.filter(i => i.includes(deferredQuery)); return ( <> <input value={query} onChange={e => setQuery(e.target.value)} /> <ul>{filtered.map(i => <li key={i}>{i}</li>)}</ul> </> ); }
useTransition vs useDeferredValue: Use useTransition when you control the state update (you own the setter). Use useDeferredValue when you receive a value as a prop or can't wrap the setter.

useImperativeHandle

useImperativeHandle(ref, createHandle, deps) lets a child component customise what a parent sees when it accesses the child's ref. Instead of exposing the entire DOM node, you expose only specific methods. Use it with forwardRef.

import { useImperativeHandle, useRef, forwardRef } from "react"; const FancyInput = forwardRef(function FancyInput(props, ref) { const inputRef = useRef(null); useImperativeHandle(ref, () => ({ // Parent can call ref.current.focus() and ref.current.clear() focus: () => inputRef.current.focus(), clear: () => { inputRef.current.value = ""; }, })); return <input ref={inputRef} {...props} />; }); // Parent — can only call focus() and clear(), nothing else function Form() { const ref = useRef(null); return ( <> <FancyInput ref={ref} /> <button onClick={() => ref.current.clear()}>Clear</button> </> ); }

useDebugValue

useDebugValue(value) displays a label for a custom hook in React DevTools. It has no effect on behaviour — it's purely for debugging. Only use it inside custom hooks, not in components directly.

import { useDebugValue, useState } from "react"; function useOnlineStatus() { const [isOnline, setIsOnline] = useState(true); // Shows "Online" or "Offline" in React DevTools for this hook useDebugValue(isOnline ? "Online" : "Offline"); return isOnline; }

use (React 19)

use(resource) lets you read the value of a Promise or Context directly inside a component. If passed a Promise, the component suspends until it resolves (requires a <Suspense> boundary above). Unlike every other hook, use can be called inside conditionals and loops.

import { use, Suspense } from "react"; // Server or parent creates the promise and passes it as a prop function UserName({ userPromise }) { const user = use(userPromise); // suspends until resolved return <p>{user.name}</p>; } // Wrap in Suspense so React knows what to show while waiting function App() { const promise = fetchUser(1); return ( <Suspense fallback={<p>Loading…</p>}> <UserName userPromise={promise} /> </Suspense> ); }
Create the promise outside the component (or with useMemo) — not inline in JSX. Creating it during render causes a new Promise every render, which re-suspends forever.

useActionState (React 19)

useActionState(action, initialState) manages state and the pending status of an async action in one hook. It returns [state, dispatch, isPending]. Every time dispatch is called with a value, the action runs and the returned state is updated — replacing the common useState + useEffect pattern for form submissions and mutations.

import { useActionState } from "react"; async function submitForm(prevState, formData) { const name = formData.get("name"); await saveName(name); return { message: `Saved: ${name}` }; } function NameForm() { const [state, dispatch, isPending] = useActionState(submitForm, { message: "" }); return ( <form action={dispatch}> <input name="name" /> <button disabled={isPending}>{isPending ? "Saving…" : "Save"}</button> {state.message && <p>{state.message}</p>} </form> ); }

useFormStatus (React 19)

useFormStatus() returns the submission status of the nearest parent <form>. Use it inside a child component (e.g. a submit button) to read pending without prop drilling. It must be called from a component that is inside the form, not the form component itself.

import { useFormStatus } from "react-dom"; // Child component — knows when the parent form is submitting function SubmitButton() { const { pending } = useFormStatus(); return <button disabled={pending}>{pending ? "Submitting…" : "Submit"}</button>; } // Parent form — no need to pass isPending as a prop function LoginForm() { return ( <form action={loginAction}> <input name="email" /> <SubmitButton /> {/* reads pending automatically */} </form> ); }

useOptimistic (React 19)

useOptimistic(state, updateFn) lets you show an immediate optimistic update to the UI while an async action (e.g. a server request) is still in flight. When the action completes, React automatically reconciles the real state. If the action fails, the optimistic value is rolled back.

import { useOptimistic } from "react"; function LikeButton({ post }) { // optimisticLikes shows +1 immediately; reverts if save fails const [optimisticLikes, addOptimisticLike] = useOptimistic( post.likes, (current) => current + 1 ); async function handleLike() { addOptimisticLike(); // instantly updates UI await likePost(post.id); // real server call } // on resolve: real state takes over return ( <button onClick={handleLike}> ❤️ {optimisticLikes} </button> ); }
useOptimistic only works inside an async transition. Call it from an action passed to useActionState, a form action, or wrapped in startTransition.

Custom Hooks

A custom hook is a JavaScript function whose name starts with use that calls other hooks. They let you extract and reuse stateful logic across multiple components without duplicating code.

// Custom hook — extract fetch logic function useFetch(url) { const [data, setData] = useState(null); const [loading, setLoading] = useState(true); const [error, setError] = useState(null); useEffect(() => { fetch(url) .then(r => r.json()) .then(setData) .catch(setError) .finally(() => setLoading(false)); }, [url]); return { data, loading, error }; } // Usage — reused in any component function Posts() { const { data, loading } = useFetch("/api/posts"); if (loading) return <p>Loading…</p>; return data.map(p => <PostCard key={p.id} {...p} />); }
Chapter 4

Component Patterns

Rendering Lists

Use .map() to render an array of items. Each item needs a stable, unique key prop so React can efficiently reconcile list changes.

function TodoList({ todos }) { return ( <ul> {todos.map(todo => ( <li key={todo.id}> {/* key must be on the outermost element */} {todo.text} </li> ))} </ul> ); }
Keys must be unique among siblings and stable across renders. Never use the array index as a key if the list can be reordered or filtered — it causes subtle bugs where React associates state with the wrong item. Use an ID from the data instead.

Conditional Rendering

// Ternary — show one or the other {isLoggedIn ? <Dashboard /> : <LoginPage />} // && — show or show nothing {hasError && <ErrorBanner message={error} />} // !! gotcha — avoid rendering 0 with && {count && <Badge />} // ❌ renders "0" if count = 0 {count > 0 && <Badge />} // ✅ explicit boolean check // Early return inside component function Profile({ user }) { if (!user) return <p>Loading…</p>; return <h1>{user.name}</h1>; }

Lifting State Up

When two sibling components need to share state, move (lift) the state to their closest common ancestor and pass it down as props. The parent owns the state; children receive it via props and communicate changes via callback props.

function Parent() { const [value, setValue] = useState(""); return ( <> {/* Input owns no state — controlled by parent */} <SearchInput value={value} onChange={setValue} /> {/* Results receive the same value */} <ResultsList query={value} /> </> ); } function SearchInput({ value, onChange }) { return <input value={value} onChange={e => onChange(e.target.value)} />; }

React.memo

React.memo is a higher-order component that memoizes a component. It skips re-rendering if the props have not changed (shallow comparison). Combine with useCallback for function props.

import { memo } from "react"; // Without memo — re-renders whenever Parent re-renders function Avatar({ src, alt }) { return <img src={src} alt={alt} />; } // With memo — only re-renders when src or alt actually change const Avatar = memo(function Avatar({ src, alt }) { return <img src={src} alt={alt} />; });
Chapter 5

Error Boundaries & Suspense

Error Boundaries

An Error Boundary is a class component that catches JavaScript errors anywhere in its child component tree during rendering or in lifecycle methods. It shows a fallback UI instead of crashing the whole app. Currently only class components can be error boundaries (or use the react-error-boundary library).

import { ErrorBoundary } from "react-error-boundary"; function Fallback({ error, resetErrorBoundary }) { return ( <div> <p>Something went wrong: {error.message}</p> <button onClick={resetErrorBoundary}>Try again</button> </div> ); } <ErrorBoundary FallbackComponent={Fallback}> <RiskyComponent /> </ErrorBoundary>

Suspense

<Suspense> lets you declaratively show a loading fallback while waiting for something to be ready — lazy-loaded components or data (with Suspense-enabled data libraries like TanStack Query v5 or Relay).

import { lazy, Suspense } from "react"; // Lazy load a component — code-splits the bundle const Dashboard = lazy(() => import("./Dashboard")); function App() { return ( <Suspense fallback={<p>Loading…</p>}> <Dashboard /> </Suspense> ); }
Chapter 6

Performance Optimization

React.memo

React.memo is a higher-order component that wraps a function component and skips re-rendering if its props haven't changed (shallow comparison). By default every parent re-render also re-renders all children — React.memo lets a child opt out of that.

import { memo } from "react"; // Without memo — re-renders every time Parent re-renders function Avatar({ src, alt }) { return <img src={src} alt={alt} />; } // With memo — only re-renders when src or alt actually change const Avatar = memo(function Avatar({ src, alt }) { return <img src={src} alt={alt} />; }); // Custom comparison — return true to SKIP re-render const Avatar = memo(AvatarFn, (prev, next) => prev.src === next.src);
memo only helps if props are stable. If a parent passes an inline object or function (style={{ color: "red" }}, onClick={() => {}}), a new reference is created every render — memo sees it as changed and re-renders anyway. Pair with useMemo / useCallback to stabilise those props.

Code Splitting with lazy & Suspense

React.lazy() lets you defer loading a component's bundle until it is first rendered. Combined with <Suspense>, React shows a fallback while the chunk loads. This keeps the initial bundle small and speeds up the first paint.

import { lazy, Suspense } from "react"; // Component loaded only when first rendered (separate JS chunk) const Dashboard = lazy(() => import("./Dashboard")); function App() { return ( <Suspense fallback={<p>Loading…</p>}> <Dashboard /> </Suspense> ); }
Split at the route level first — lazy-loading each page gives the biggest gains for the least complexity. Only go deeper (splitting individual heavy components) if profiling shows it's worth it.

List Virtualisation

Rendering thousands of DOM nodes at once is slow. Virtualisation renders only the items currently visible in the viewport, recycling nodes as the user scrolls. Use react-window or @tanstack/react-virtual for long lists.

import { FixedSizeList } from "react-window"; function Row({ index, style }) { return <div style={style}>Row {index}</div>; } function BigList() { return ( <FixedSizeList height={400} itemCount={10000} itemSize={35} width="100%" > {Row} </FixedSizeList> ); }

Avoid Unnecessary Re-renders

Most performance problems in React are caused by components re-rendering more often than needed. Common causes and fixes:

CauseFix
New object/array literal in JSXMove outside component or wrap in useMemo
Inline function as propWrap handler in useCallback
Context value changes every renderMemoize the value object with useMemo
Child always re-renders with parentWrap child in React.memo
Expensive computation on every renderCache result with useMemo
State too high up the treeColocate state — move it down to the component that owns it
// ❌ New object every render — Context consumers always re-render <UserContext.Provider value={{ user, logout }}> // ✅ Memoize the value — consumers only re-render when user changes const value = useMemo(() => ({ user, logout }), [user]); <UserContext.Provider value={value}>

Profiling with React DevTools

The React DevTools Profiler records a render timeline showing which components rendered, how long each took, and why they rendered. Use it to find actual bottlenecks before reaching for optimisation tools.

StepWhat to do
1. Open DevToolsInstall React DevTools browser extension → open the Profiler tab.
2. RecordClick Record, interact with the slow part of the UI, then stop.
3. Inspect flamegraphEach bar is a component render. Tall/wide bars took longest. Grey = not re-rendered.
4. Check "why did this render?"Enable "Record why each component rendered" in settings. Shows prop/state/context changes.
Profile before optimising. Adding memo, useMemo, and useCallback everywhere adds cognitive overhead and can hurt performance (memoisation has its own cost). Only apply them where the Profiler shows a real problem.
