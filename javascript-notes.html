Chapter 1

Introduction to JavaScript

What is JavaScript?

JavaScript is a high-level, interpreted programming language that runs in the browser (and on servers via Node.js). It adds interactivity, logic, and behaviour to web pages.

  • Interpreted — code is executed line-by-line at runtime, not compiled ahead of time.
  • Dynamically typed — variable types are determined at runtime, not declared in advance.
  • Single-threaded — runs one operation at a time, but handles async work via the event loop.
HTML / CSS / JS roles: HTML = structure, CSS = presentation, JavaScript = behaviour. JS can read and manipulate both HTML and CSS through the DOM.

Hoisting

Hoisting is JavaScript's behaviour of moving variable and function declarations to the top of their scope during the compilation phase, before code runs.

// var — hoisted and initialised to undefined console.log(x); // undefined (no error) var x = 5; // What the engine actually sees: var x; // declaration hoisted console.log(x); // undefined x = 5; // assignment stays here // Function declaration — fully hoisted (body too) greet(); // "Hello" — works before definition function greet() { console.log("Hello"); } // let / const — hoisted but NOT initialised → TDZ console.log(y); // ❌ ReferenceError let y = 10;

Temporal Dead Zone (TDZ)

The Temporal Dead Zone is the period between the start of the block scope and the point where let or const is declared. Accessing the variable in this zone throws a ReferenceError.

{ // TDZ for `color` starts here ↓ console.log(color); // ❌ ReferenceError: Cannot access before init let color = "red"; // TDZ ends here ↑ console.log(color); // ✅ "red" }
Why TDZ exists: It prevents reading a variable before it has a meaningful value — a class of bug that var's silent undefined hoisting caused. Always declare variables at the top of their block to avoid TDZ issues.

Rule of thumb: Use const by default. Use let when the value must be reassigned. Never use var.

Primitive vs Reference types:
Primitives are copied by value — let b = a gives b its own copy.

// Common truthy gotchas if ([]) { } // ✅ empty array is truthy if ({}) { } // ✅ empty object is truthy if ("0") { } // ✅ non-empty string is truthy // Use ?? not || when 0 or "" are valid values const count = userCount || 10; // ❌ 0 becomes 10 const count = userCount ?? 10; // ✅ 0 stays 0

Type Coercion & Equality

Type coercion is JavaScript automatically converting one type to another when an operation requires it.

// Implicit coercion examples "5" + 3 // "53" — number coerced to string (+ prefers string) "5" - 3 // 2 — string coerced to number (- is numeric only) true + 1 // 2 — true coerced to 1 false + 1 // 1 — false coerced to 0 // == loose equality — coerces types before comparing 0 == "" // true ← dangerous 0 == false // true null == undefined // true (only these two equal each other loosely) "1" == 1 // true // === strict equality — checks type AND value, no coercion 0 === "" // false ✅ 1 === "1" // false ✅
Always use ===. The only acceptable use of == is val == null as a shorthand to check for both null and undefined at once.

Parameters — Defaults, Rest & Spread

Closures

A closure is a function that remembers the variables from its outer (lexical) scope even after that outer function has finished executing. The inner function keeps a live reference — not a copy.

function makeCounter() { let count = 0; // lives in makeCounter's scope return { increment() { count++; }, decrement() { count--; }, value() { return count; } }; } const counter = makeCounter(); // makeCounter has finished — but count lives on counter.increment(); // count = 1 counter.increment(); // count = 2 counter.value(); // 2 // Each call to makeCounter() creates a new independent closure const c2 = makeCounter(); c2.increment(); // c2's count = 1 (separate from counter)
Common use cases for closures: private variables (as shown above), factory functions, memoization, and event listeners that capture loop variables. Closures are one of the most powerful and frequently-used patterns in JavaScript.

The this Keyword

this refers to the object that is currently calling the function. Its value is determined by how the function is called, not where it is defined — except in arrow functions, which capture this from their surrounding lexical scope.

new Fn()The newly created object instance

Chapter 7

Async JavaScript

The Event Loop

JavaScript is single-threaded — it runs one operation at a time. The event loop is what allows it to handle async work (timers, network) without blocking.

ComponentWhat it does
Call stackWhere synchronous code executes. One frame per function call. LIFO.
Web APIsBrowser-provided async capabilities: setTimeout, fetch, DOM events. Run outside the JS thread.
Microtask queueHolds resolved Promise callbacks (.then, await, queueMicrotask). Drains completely before the next task.
Task queue (macrotask)Holds setTimeout, setInterval, I/O callbacks. One task per event loop tick.
Event loopChecks: if call stack is empty → drain microtask queue → pick one macrotask → repeat.
console.log("1"); // sync — runs first setTimeout(() => console.log("2"), 0); // macrotask — runs last Promise.resolve().then(() => console.log("3")); // microtask — before macrotask console.log("4"); // sync — runs second // Output: 1, 4, 3, 2 // Microtasks (Promise) always run before macrotasks (setTimeout)
Microtasks before macrotasks: Promise callbacks are microtasks and always run before setTimeout/setInterval callbacks, even if the timer delay is 0. The microtask queue is fully drained after every task before the next macrotask starts.

Promises

A Promise is an object representing an asynchronous operation. It is in one of three states:

StateMeaningTransitions to
pendingInitial state — neither fulfilled nor rejected.fulfilled or rejected
fulfilledOperation succeeded. resolve(value) was called.Final (settled)
rejectedOperation failed. reject(reason) was called.Final (settled)
// Creating a promise const p = new Promise((resolve, reject) => { const ok = true; if (ok) resolve("data"); else reject(new Error("failed")); }); // Consuming with .then / .catch / .finally p .then(data => console.log(data)) // runs on fulfil .catch(err => console.error(err)) // runs on reject .finally(() => console.log("done")); // always runs // .then returns a new promise — chains are possible fetch("/api") .then(res => res.json()) .then(data => console.log(data)) .catch(err => console.error(err));

Promise Combinators

When you need to run multiple async operations together:

MethodResolves whenRejects whenUse for
Promise.all(arr)All resolveAny one rejects (fast-fail)Parallel tasks where all must succeed
Promise.allSettled(arr)All settle (any outcome)NeverRun all, collect all results regardless of failure
Promise.race(arr)First to settle (win or fail)First to rejectTimeout patterns
Promise.any(arr)First to fulfilAll rejectFirst success wins
// Promise.all — parallel, fast-fail const [user, posts] = await Promise.all([ fetch("/api/user").then(r => r.json()), fetch("/api/posts").then(r => r.json()), ]); // Promise.allSettled — get all results even if some fail const results = await Promise.allSettled([p1, p2, p3]); results.forEach(r => { if (r.status === "fulfilled") console.log(r.value); if (r.status === "rejected") console.error(r.reason); });

async / await

async/await is syntactic sugar over Promises. async makes a function always return a Promise. await pauses execution inside the async function until the Promise settles — the rest of the program continues unblocked.

async function getUser(id) { try { const res = await fetch(`/api/users/${id}`); // pauses here if (!res.ok) throw new Error(`HTTP ${res.status}`); const data = await res.json(); // pauses again return data; } catch (err) { console.error(err); throw err; // re-throw so caller knows it failed } } // await in parallel — don't await sequentially if not needed // ❌ Sequential (slow — waits for each before starting next) const a = await fetchA(); const b = await fetchB(); // ✅ Parallel (fast — both start at the same time) const [a, b] = await Promise.all([fetchA(), fetchB()]);
Always use try/catch with await — unhandled rejected promises cause runtime errors. An async function always returns a Promise, so the caller can also chain .catch() on it. Re-throw errors if the caller needs to know about them.

Fetch API

// GET request const res = await fetch("https://api.example.com/data"); if (!res.ok) throw new Error(`Error: ${res.status}`); const data = await res.json(); // POST request const res2 = await fetch("/api/users", { method: "POST", headers: { "Content-Type": "application/json" }, body: JSON.stringify({ name: "Alice" }) }); const created = await res2.json();
fetch only rejects on network errors (e.g. no internet). A 404 or 500 HTTP response still resolves the promise — it does NOT trigger .catch. Always check res.ok (true for 200–299) or res.status to handle HTTP-level errors explicitly.

Classes & Prototypes

Prototype Chain

Every JavaScript object has an internal link to another object called its prototype. When you access a property, JS looks on the object first, then walks up the prototype chain until it finds it or reaches null.

// Every object inherits from Object.prototype const arr = [1, 2, 3]; // arr → Array.prototype → Object.prototype → null arr.push // found on Array.prototype arr.toString // found on Object.prototype (walked up the chain) // Object.getPrototypeOf() reveals the chain Object.getPrototypeOf(arr) === Array.prototype; // true // hasOwnProperty — checks only the object itself, not the chain arr.hasOwnProperty("length"); // true — own property arr.hasOwnProperty("push"); // false — inherited from Array.prototype
Classes are syntactic sugar over prototypes. When you write class Dog extends Animal, JS sets Dog.prototype's prototype to Animal.prototype. Methods defined in a class body are placed on the prototype — not on each instance — so they're shared and memory-efficient.

Classes

class Animal { constructor(name, sound) { this.name = name; // instance property — on each object this.sound = sound; } speak() { // prototype method — shared across all instances return `${this.name} says ${this.sound}.`; } } class Dog extends Animal { constructor(name) { super(name, "woof"); // must call super() before using this this.tricks = []; } learn(trick) { this.tricks.push(trick); } speak() { return `${super.speak()} Knows: ${this.tricks.join(", ")}`; } } const d = new Dog("Rex"); d.learn("sit"); d.speak(); // "Rex says woof. Knows: sit" d instanceof Dog; // true d instanceof Animal; // true — inherited

Error Handling

try / catch / finally

try wraps code that may throw. catch receives the thrown value (always an Error by convention). finally runs regardless of success or failure — ideal for cleanup.

try { const data = JSON.parse(badJson); // throws SyntaxError on bad input processData(data); } catch (err) { // err.name — type of error e.g. "SyntaxError" // err.message — human-readable description // err.stack — stack trace (helpful for debugging) console.error(err.name, err.message); } finally { closeConnection(); // always runs — even if try returns or throws } // Catch specific error types try { riskyOp(); } catch (err) { if (err instanceof TypeError) { console.error("Type problem:", err.message); } else { throw err; // re-throw anything you can't handle } }

Custom Errors & Throwing

Map & Set

Map

Key-value pairs. Keys can be any type (objects, functions, primitives). Maintains insertion order. Better than plain objects for dynamic key collections.

Set

Collection of unique values. Deduplicates automatically. Maintains insertion order. Has O(1) lookup via .has().

// Map — any key type, no prototype pollution risk const map = new Map(); map.set("name", "Alice"); map.set(userObj, "admin"); // object as key — impossible with plain obj map.get("name"); // "Alice" map.has("name"); // true map.size; // 2 map.delete("name"); for (const [k, v] of map) { console.log(k, v); } // Set — unique values const set = new Set([1, 2, 2, 3]); // {1, 2, 3} set.add(4); set.has(2); // true set.delete(1); set.size; // 3 // Remove duplicates from array — classic use case const unique = [...new Set([1, 2, 2, 3, 3])]; // [1, 2, 3] // Set intersection, union, difference const a = new Set([1, 2, 3]); const b = new Set([2, 3, 4]); const union = new Set([...a, ...b]); // {1,2,3,4} const intersection = new Set([...a].filter(x => b.has(x))); // {2,3}
Map vs Object: Use Map when keys are dynamic, non-string, or when key count needs to be tracked easily via .size. Objects are fine for static, string-keyed data structures.

Symbols

A Symbol is a guaranteed-unique, immutable primitive value. Two symbols created with the same description are never equal. Symbols are commonly used as unique property keys that won't clash with other code.

const id1 = Symbol("id"); const id2 = Symbol("id"); id1 === id2; // false — always unique // As object key — won't clash with other property names const SECRET = Symbol("secret"); const obj = { [SECRET]: 42, name: "Alice" }; obj[SECRET]; // 42 Object.keys(obj); // ["name"] — Symbol keys are hidden from enumeration // Well-known symbols — customise built-in behaviour class Range { [Symbol.iterator]() { // makes the object iterable with for...of let i = 0; return { next: () => ({ value: i++, done: i > 3 }) }; } } [...new Range()]; // [0, 1, 2]
