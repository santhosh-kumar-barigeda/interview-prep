Chapter 1

Introduction to Next.js

What is Next.js?

Next.js is a React framework by Vercel that adds server-side capabilities to React — routing, data fetching, rendering strategies, image optimisation, and API routes — all with zero config. It is the most widely used full-stack React framework.

FeatureWhat it gives you
File-based routingEvery file in app/ automatically becomes a route
Server ComponentsRender on the server by default — no JS sent to the client
Data fetchingfetch with built-in caching and revalidation
API routesBackend endpoints inside the same project
Image optimisation<Image> component with lazy loading, WebP, resizing
Fonts & metadataBuilt-in font loading and <head> metadata API
MiddlewareEdge functions that run before every request

App Router vs Pages Router

Next.js has two routing systems. The App Router (introduced in Next.js 13, stable in 14) is the modern default. The Pages Router is the legacy system — still supported but new projects should use App Router.

App Router (app/)Pages Router (pages/)
Default sinceNext.js 13 (stable 14)Next.js 1–12
Rendering defaultServer ComponentsClient Components
Data fetchingfetch in Server Components, usegetServerSideProps, getStaticProps
LayoutsNested layout.tsx files_app.tsx + custom wrappers
Loading statesloading.tsx filesManual
Error handlingerror.tsx filesManual error boundaries
All notes here cover the App Router. Use it for all new projects.

Chapter 2

Routing

File-based Routing

Every folder inside app/ maps to a URL segment. A page.tsx file inside that folder makes the route publicly accessible. Folders without a page.tsx are private — they can hold layouts, components, and utilities.

File pathURL
app/page.tsx/
app/about/page.tsx/about
app/blog/[slug]/page.tsx/blog/:slug
app/shop/[...slug]/page.tsx/shop/* (catch-all)
app/(marketing)/page.tsx/ (route group — no URL segment)
app/@modal/page.tsxParallel route slot

Dynamic Routes

Wrap a folder name in [brackets] to make it dynamic. The segment value is available in the params prop of the page component.

// app/blog/[slug]/page.tsx type Props = { params: { slug: string } }; export default async function BlogPost({ params }: Props) { const post = await fetchPost(params.slug); return <article>{post.title}</article>; } // Generate static pages at build time export async function generateStaticParams() { const posts = await fetchAllPosts(); return posts.map(p => ({ slug: p.slug })); }

Layouts

A layout.tsx file wraps all pages in its folder and below. Layouts are persistent — they don't remount when navigating between child routes. The root layout (app/layout.tsx) must include <html> and <body> tags.

// app/layout.tsx — root layout export default function RootLayout({ children }: { children: React.ReactNode }) { return ( <html lang="en"> <body> <Navbar /> {children} <Footer /> </body> </html> ); } // app/dashboard/layout.tsx — nested layout export default function DashboardLayout({ children }: { children: React.ReactNode }) { return ( <div className="dashboard"> <Sidebar /> <main>{children}</main> </div> ); }

Special Files

FilePurpose
page.tsxUI for a route — makes the segment publicly accessible.
layout.tsxShared UI that wraps a segment and all its children. Does not remount on navigation.
loading.tsxInstant loading UI shown via Suspense while the page or data loads.
error.tsxError boundary for a segment. Receives error and reset props. Must be a Client Component.
not-found.tsxUI shown when notFound() is called or no route matches.
route.tsAPI route handler — exports GET, POST, etc. functions.
middleware.tsRuns at the edge before every request.
template.tsxLike layout but remounts on every navigation (creates a new instance).

Navigation

Use <Link> for declarative navigation — it prefetches routes in the viewport automatically. Use useRouter for programmatic navigation in Client Components.

import Link from "next/link"; import { useRouter } from "next/navigation"; // Declarative — use this by default <Link href="/dashboard">Dashboard</Link> <Link href={`/blog/${slug}`}>Post</Link> // Programmatic — in Client Components only const router = useRouter(); router.push("/dashboard"); router.replace("/login"); // no history entry router.back(); // Read current path / params / search params import { usePathname, useSearchParams } from "next/navigation"; const pathname = usePathname(); // "/dashboard/settings" const searchParams = useSearchParams(); const query = searchParams.get("q");
Chapter 3

Server & Client Components

Server Components

Server Components (RSC) are the default in the App Router. They render exclusively on the server — their code is never sent to the browser. This means you can directly access databases, file systems, and secrets without an API layer.

Can do

Async/await at the top level, direct DB queries, read server-only env vars, import heavy server libraries

Cannot do

Use hooks (useState, useEffect), browser APIs, event handlers, anything requiring interactivity

// app/dashboard/page.tsx — Server Component (default) import { db } from "@/lib/db"; export default async function DashboardPage() { const stats = await db.getStats(); // direct DB call — no API needed return ( <main> <h1>Dashboard</h1> <StatsCard data={stats} /> </main> ); }

Client Components

Add "use client" at the top of a file to mark it as a Client Component. It renders on the server first (for the initial HTML), then hydrates in the browser. Use Client Components for anything interactive — state, effects, event handlers, browser APIs.

"use client"; import { useState } from "react"; export function Counter() { const [count, setCount] = useState(0); return ( <button onClick={() => setCount(c => c + 1)}> Count: {count} </button> ); }
Push Client Components to the leaves. Keep the top-level pages and layouts as Server Components and only mark the interactive parts as "use client". This minimises the JS bundle sent to the browser.

Composition Pattern

Server Components can import and render Client Components. Client Components cannot import Server Components — but they can receive Server Components as children props.

// ✅ Server Component passes RSC as children to a Client Component import { Modal } from "./Modal"; // Client Component import { UserList } from "./UserList"; // Server Component export default function Page() { return ( <Modal> <UserList /> {/* rendered on server, passed as children */} </Modal> ); } // ❌ Cannot import a Server Component inside a Client Component file "use client"; import { UserList } from "./UserList"; // Error — RSC in client boundary
Chapter 4

Data Fetching

fetch in Server Components

Next.js extends the native fetch API with caching and revalidation options. Fetch calls in Server Components are automatically deduplicated — the same URL fetched multiple times in one render is only requested once.

// Static — cached indefinitely (like getStaticProps) const data = await fetch("https://api.example.com/posts", { cache: "force-cache", // default behaviour }); // Dynamic — never cached (like getServerSideProps) const data = await fetch("https://api.example.com/posts", { cache: "no-store", }); // ISR — revalidate every N seconds const data = await fetch("https://api.example.com/posts", { next: { revalidate: 60 }, // refresh cache every 60s });

Route Segment Config

Export config variables from a page or layout to control caching for the entire route segment.

// app/dashboard/page.tsx export const dynamic = "force-dynamic"; // always render fresh (SSR) export const dynamic = "force-static"; // always static (SSG) export const revalidate = 3600; // ISR — revalidate every hour export const runtime = "edge"; // run on Edge Runtime

Parallel & Sequential Fetching

Await multiple fetches in sequence (waterfall) or in parallel with Promise.all. Always prefer parallel unless each request depends on the previous one.

// ❌ Sequential — waterfall — each waits for the previous const user = await fetchUser(id); const posts = await fetchPosts(user.id); // waits for user first // ✅ Parallel — both fire at the same time const [user, posts] = await Promise.all([ fetchUser(id), fetchPosts(id), ]);

Streaming with Suspense

Wrap slow data-fetching components in <Suspense> to stream them progressively — the shell of the page arrives first, then each suspended part streams in as it resolves. A loading.tsx file does this automatically for the entire page.

import { Suspense } from "react"; import { UserPosts } from "./UserPosts"; export default function Page() { return ( <main> <h1>Profile</h1> <Suspense fallback={<p>Loading posts…</p>}> <UserPosts /> {/* streams in when data is ready */} </Suspense> </main> ); }

Server Actions

Server Actions are async functions that run on the server and can be called directly from Client or Server Components — no API route needed. Mark a function with "use server" to make it a Server Action.

// app/actions.ts "use server"; import { revalidatePath } from "next/cache"; import { db } from "@/lib/db"; export async function createPost(formData: FormData) { const title = formData.get("title") as string; await db.posts.create({ data: { title } }); revalidatePath("/posts"); // purge cache for /posts } // Use in a form — no onSubmit needed <form action={createPost}> <input name="title" /> <button>Create</button> </form> // Or call directly from a Client Component import { createPost } from "@/app/actions"; const [state, dispatch, isPending] = useActionState(createPost, null);
Chapter 5

Rendering Strategies

Static, Dynamic & ISR

Next.js automatically determines the rendering strategy per route based on the data-fetching options you use. You can override it with the route segment config.

StrategyWhenUse for
Static (SSG)Built once at build time. Served from CDN.Marketing pages, blogs, docs — content that rarely changes.
ISRStatic, but revalidated on a schedule or on demand.E-commerce product pages, dashboards — content that changes but not on every request.
Dynamic (SSR)Rendered on every request.Personalised pages, auth-gated content, real-time data.
Client-sideRendered in the browser after hydration.Highly interactive UIs, user-specific data after login.
A route becomes dynamic automatically if it uses: cookies(), headers(), searchParams, cache: "no-store", or noStore().

On-demand Revalidation

Purge cached data without waiting for the revalidation interval using revalidatePath or revalidateTag. Typically called from a Server Action or API route after a mutation.

import { revalidatePath, revalidateTag } from "next/cache"; // Revalidate a specific URL path revalidatePath("/posts"); revalidatePath("/posts/[slug]", "page"); // Tag fetches and revalidate by tag const data = await fetch("/api/posts", { next: { tags: ["posts"] } }); revalidateTag("posts"); // purges all fetches tagged "posts"
Chapter 6

API Routes

Route Handlers

Create route.ts files inside app/ to define API endpoints. Export named functions matching HTTP methods: GET, POST, PUT, PATCH, DELETE.

// app/api/posts/route.ts import { NextRequest, NextResponse } from "next/server"; export async function GET(req: NextRequest) { const posts = await db.getPosts(); return NextResponse.json(posts); } export async function POST(req: NextRequest) { const body = await req.json(); const post = await db.createPost(body); return NextResponse.json(post, { status: 201 }); }

Dynamic API Routes

// app/api/posts/[id]/route.ts export async function GET( req: NextRequest, { params }: { params: { id: string } } ) { const post = await db.getPost(params.id); if (!post) return NextResponse.json({ error: "Not found" }, { status: 404 }); return NextResponse.json(post); } export async function DELETE( req: NextRequest, { params }: { params: { id: string } } ) { await db.deletePost(params.id); return new Response(null, { status: 204 }); }

Reading Request Data

export async function GET(req: NextRequest) { // Search params — /api/search?q=hello const q = req.nextUrl.searchParams.get("q"); // Headers const token = req.headers.get("authorization"); // Cookies const session = req.cookies.get("session"); } export async function POST(req: NextRequest) { // JSON body const body = await req.json(); // Form data const formData = await req.formData(); }
Chapter 7

Middleware

What is Middleware?

Middleware runs at the Edge before a request is processed — before the page or API route handler. Use it for auth checks, redirects, rewrites, geolocation, and A/B testing. Create a middleware.ts file at the project root (next to app/).

// middleware.ts import { NextRequest, NextResponse } from "next/server"; export function middleware(req: NextRequest) { const token = req.cookies.get("session"); // Redirect to login if unauthenticated if (!token) { return NextResponse.redirect(new URL("/login", req.url)); } return NextResponse.next(); // continue to the route } // Only run middleware on these paths export const config = { matcher: ["/dashboard/:path*", "/api/:path*"], };
Middleware runs on the Edge Runtime — it cannot use Node.js APIs. Keep it lightweight. For heavy auth logic, verify tokens (JWT) in middleware but do the full session lookup in the route handler.
Chapter 8

Metadata & SEO

Static Metadata

Export a metadata object from any page.tsx or layout.tsx to set <head> tags. Next.js merges metadata from parent layouts down to the page.

import type { Metadata } from "next"; export const metadata: Metadata = { title: "My App", description: "The best app ever", openGraph: { title: "My App", description: "The best app ever", images: ["/og-image.png"], }, twitter: { card: "summary_large_image" }, };

Dynamic Metadata

Export a generateMetadata async function to compute metadata from route params or fetched data.

import type { Metadata } from "next"; export async function generateMetadata( { params }: { params: { slug: string } } ): Promise<Metadata> { const post = await fetchPost(params.slug); return { title: post.title, description: post.excerpt, openGraph: { images: [post.coverImage] }, }; }
Chapter 9

Optimisation

Image Optimisation

The <Image> component from next/image automatically optimises images — lazy loading, modern formats (WebP/AVIF), responsive sizes, and prevents layout shift.

import Image from "next/image"; // Fixed size <Image src="/avatar.jpg" alt="Avatar" width={64} height={64} /> // Fill container — parent must be position: relative <div style={{ position: "relative", width: "100%", height: "400px" }}> <Image src="/hero.jpg" alt="Hero" fill className="object-cover" /> </div> // Prioritise above-the-fold images (disables lazy loading) <Image src="/hero.jpg" alt="Hero" width={1200} height={600} priority />

Font Optimisation

next/font loads Google Fonts and local fonts at build time — zero layout shift, no external network requests at runtime, automatic self-hosting.

import { Inter, DM_Mono } from "next/font/google"; const inter = Inter({ subsets: ["latin"], variable: "--font-inter", }); export default function RootLayout({ children }) { return ( <html className={inter.variable}> <body>{children}</body> </html> ); }

Script Optimisation

Use next/script to control when third-party scripts load without blocking rendering.

import Script from "next/script"; // afterInteractive (default) — loads after page is interactive <Script src="https://analytics.example.com/script.js" /> // lazyOnload — loads during idle time <Script src="https://chat-widget.com/widget.js" strategy="lazyOnload" /> // beforeInteractive — loads before hydration (use sparingly) <Script src="/critical.js" strategy="beforeInteractive" />
